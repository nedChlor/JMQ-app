# Техническое задание
## Мобильное приложение для сервисного центра (JMQ) — MVP на модели J7

Версия: 1.0
Область: MVP на одной модели автомобиля (JAC J7). Архитектура должна допускать
последующее расширение на остальные ~14 моделей без переделки основ.

---

## 1. Назначение и цель

Мобильное приложение для автомехаников сервисного центра, обслуживающего
автомобили JAC. Приложение даёт быстрый доступ к технической документации
(руководства по ремонту и обслуживанию, электросхемы, обучающие материалы)
и к расшифровке кодов неисправностей (DTC), с возможностью найти нужный
документ по коду ошибки, по категории системы автомобиля, или по
ключевым словам.

Основная проблема, которую решает приложение: механик тратит время на
поиск нужного PDF среди десятков файлов и на пролистывание длинных
документов вручную. Приложение должно резко сократить время от
"возникла проблема / известен код ошибки" до "нашёл нужный документ".

### Пользователи
Автомеханики сервисного центра, использующие личные смартфоны
(Android в первую очередь, iOS желательно позже). Технический уровень
пользователей — не предполагается уверенное владение сложным ПО,
интерфейс должен быть максимально простым и быстрым в использовании
в условиях цеха (может быть шумно, руки в перчатках/грязные, нет
времени на долгую навигацию).

### Не входит в MVP (сознательно исключено)
- Разбивка документов на смысловые разделы (sections) — этот функционал
  ранее прорабатывался, но решено отменить и переделать отдельным
  инструментом позже. **В MVP документ = один файл целиком, без
  внутреннего деления на разделы.**
- iOS-сборка (архитектура не должна это исключать, но приоритет — Android).
- Оффлайн-режим (MVP работает через сеть, файлы и база — в Firebase).
- Поддержка остальных 14 моделей автомобилей (архитектура должна закладывать
  такую возможность, но данные и контент — только J7).
- Многоязычность интерфейса (только русский).
- Авторизация/разграничение прав пользователей (открытый доступ всем,
  кто установил приложение — можно добавить позже).

## 2. Общая архитектура

```
┌─────────────────────┐
│   Flutter-приложение │  (Android, позже iOS)
│   (клиент)           │
└──────────┬───────────┘
           │
           │  Firestore SDK / Storage SDK
           ▼
┌──────────────────────────────────────┐
│              Firebase                 │
│  ┌────────────┐     ┌──────────────┐ │
│  │ Firestore  │     │   Storage    │ │
│  │ (данные)   │     │   (файлы)    │ │
│  └────────────┘     └──────────────┘ │
└────────────────────────────────────────┘
```

- **Firestore** — база данных: коды ошибок (`dtc_codes`), метаданные
  документов (`documents`), справочник категорий (`categories`).
- **Firebase Storage** — сами файлы документации в формате PDF (все
  документы, независимо от исходного формата, должны быть
  сконвертированы в PDF на этапе подготовки данных — см. раздел 4).
- **Flutter-приложение** — тонкий клиент, вся логика — чтение из
  Firestore/Storage и отображение. Никакой сложной бизнес-логики на
  клиенте не требуется для MVP.
- **State management:** `flutter_riverpod` (актуальный подход с
  `riverpod_generator` / аннотацией `@riverpod`, а не устаревший
  `StateNotifierProvider` вручную и не пакет `provider`).

### Почему не свой backend
Для MVP на одну модель отдельный backend-сервер избыточен: Firestore
и Storage полностью покрывают потребности (хранение структурированных
данных + файлов), не требуют администрирования сервера, бесплатны в
пределах разумных объёмов на старте. Полнотекстовый поиск в MVP
реализуется на стороне клиента (см. раздел 6), поэтому отдельный
поисковый сервис (Algolia/Typesense/Elasticsearch) не требуется на
этом этапе.

## 3. Модель данных (Firestore)

### 3.1. Коллекция `dtc_codes`

Один документ Firestore = один код неисправности.

**ID документа:** `{model}_{ecu}_{dtc_code}`, например `J7_EMS_P000A00`.
(Используется составной ID, а не автогенерируемый, чтобы искать код
напрямую через `getDoc`, без query.)

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `model` | string | Модель автомобиля, например `"J7"` |
| `ecu` | string | Код электронного блока управления, например `"EMS"`, `"BCM"`, `"TCU"` |
| `category` | string | Путь категории документации, соответствующей этому ECU, например `"01_Engine"` или `"04_Body/02_Electrical"` |
| `dtc_code` | string | Код неисправности, например `"P000A00"` |
| `description_en` | string | Описание кода на английском (из исходной таблицы) |
| `description_ru` | string | Описание на русском (перевод/адаптация — если на момент разработки перевод не готов, дублировать `description_en` и пометить полем `needs_translation: true`) |

**Индексы:** составной индекс не обязателен для MVP, так как основной
сценарий — прямой доступ по ID документа. Но для сценария "показать
все коды по модели" или "все коды по ECU" нужен обычный индекс на
поля `model` и `ecu` (Firestore создаёт автоматические одно-полевые
индексы по умолчанию — этого достаточно).

**Источник данных:** уже подготовленный JSON-массив с 1227 кодами J7
(см. файл, переданный ранее — `j7_dtc_codes.json`), готовый к импорту
через отдельный одноразовый скрипт-импортёр (Node.js или Python с
Firebase Admin SDK).

### 3.2. Коллекция `documents`

Один документ Firestore = один файл документации (после конвертации
в PDF, если исходник был DOCX/PPTX).

**ID документа:** произвольный, например автогенерируемый Firestore ID,
либо осмысленный slug (`j7_01_engine_hfc4gb24d_maintenance`).

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `model` | string | `"J7"` |
| `category` | string | Путь категории, соответствующий структуре папок, например `"01_Engine/HFC4GB2.4D"` |
| `category_display_ru` | string | Человекочитаемое название категории на русском для UI, например `"Двигатель → HFC4GB2.4D"` |
| `title` | string | Название документа на русском, например `"Двигатель HFC4GB2.4D 1.5T — Руководство по обслуживанию"` |
| `ecu_tags` | array\<string\> | Список ECU, к которым относится документ, например `["EMS", "FPCM"]` — используется для связи "код ошибки → документ" |
| `file_path` | string | Путь к файлу в Firebase Storage, например `"j7/01_Engine/HFC4GB2.4D/maintenance.pdf"` |
| `file_size_bytes` | number | Размер файла (для отображения пользователю перед скачиванием — полезно на слабом интернете) |
| `page_count` | number | Количество страниц (если легко достать при подготовке данных) |
| `source_format` | string | Исходный формат до конвертации: `"pdf"`, `"docx"`, `"pptx"` — для внутренней информации/отладки, не показывается пользователю |
| `full_text` | string | Полный извлечённый текст документа — используется для полнотекстового поиска на клиенте (см. раздел 6). Если текст очень большой (десятки тысяч символов), рассмотреть хранение в отдельном документе/поддокументе или в Storage как отдельный .txt файл, подгружаемый по требованию — см. примечание в разделе 6.3 |

### 3.3. Коллекция `categories`

Справочник категорий для построения навигации в UI (соответствует
структуре папок документации).

**ID документа:** путь категории, например `01_Engine`,
`04_Body_01_Structure` (слэши в Firestore ID лучше заменить на
подчёркивания или использовать вложенную структуру — решить при
реализации).

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `model` | string | `"J7"` |
| `path` | string | Оригинальный путь категории, например `"04_Body/02_Electrical"` |
| `display_name_ru` | string | Название для UI, например `"Кузов → Электрооборудование"` |
| `parent_path` | string \| null | Путь родительской категории, для построения дерева навигации (`null` для категорий верхнего уровня) |
| `sort_order` | number | Порядок отображения (можно взять из числового префикса папки: `01_Engine` → `1`) |

## 4. Подготовка данных (уже частично выполнено)

Этот раздел фиксирует то, что уже сделано и считается готовым входом
для разработки приложения.

### 4.1. Готово на момент начала разработки приложения
- Таблица кодов неисправностей J7 разобрана в чистый JSON
  (1227 записей, поля: model, ecu, category, dtc_code, description_en,
  description_cn).
- Все документы J7 приведены к единому формату PDF (исходные DOCX/PPTX
  сконвертированы через LibreOffice headless, качество конвертации
  проверено вручную — картинки, таблицы, кириллица отображаются
  корректно).
- Структура папок документации зафиксирована (см. раздел 4.2).
- Из документов извлечён полный текст (для полнотекстового поиска).

### 4.2. Структура категорий документации J7

```
01_Engine
├── HFC4GB2.4D
└── HFC4GB2.4E
02_Transmission
├── CVT_Punch
└── MF622D35
03_Chassis
04_Body
├── 01_Structure
└── 02_Electrical
05_Electrical_Systems
├── 01_Circuit_Diagrams
├── 02_AVM
├── 03_TPMS
└── 04_MP5 (пусто на диске — соответствующий контент физически лежит в 99_Firmware_Updates/MP5, категоризация по смыслу, не по физическому пути)
06_Calibration
99_Firmware_Updates
└── MP5
```

### 4.3. Маппинг ECU → категория (для связи кодов ошибок с документами)

| ECU | Категория |
|---|---|
| EMS, FPCM | `01_Engine` |
| TCU | `02_Transmission` |
| ESP, EPS, ESCL | `03_Chassis` |
| BCM, SRS | `04_Body/02_Electrical` |
| AVM, PDC | `05_Electrical_Systems/02_AVM` |
| PLG | `05_Electrical_Systems` |
| MP5, ICM (все варианты) | `05_Electrical_Systems/04_MP5` |
| HVAC (оба варианта), Tbox, APM, ACC, APA, BSD, MPC, AMB, WLC | `05_Electrical_Systems` (без более точной подкатегории на момент MVP) |

Примечание: этот маппинг черновой и может уточняться по мере того,
как будет ясно, в каком именно документе на практике описана та или
иная система (особенно для группы ADAS/климат-контроль, у которых
пока нет точной подкатегории).

### 4.4. Что нужно доделать перед импортом в Firestore
- Скрипт-импортёр (Python или Node.js, с использованием Firebase Admin
  SDK), который:
  1. Читает JSON с кодами ошибок → создаёт документы в `dtc_codes`.
  2. Читает каталог документов (метаданные + пути к файлам) → создаёт
     документы в `documents`, загружает сами PDF-файлы в Firebase
     Storage по соответствующим путям.
  3. Создаёт документы в `categories` на основе структуры папок.
- Присвоить каждому документу человекочитаемый `title` на русском
  (если ещё не сделано) — например через одноразовую обработку
  DeepSeek API по названиям файлов и первым страницам, либо вручную,
  раз документов всего 21.

## 5. Экраны приложения

### 5.1. Главный экран

Простой выбор из двух основных сценариев:
- Кнопка/раздел "Поиск по коду ошибки"
- Кнопка/раздел "Категории документации" (навигация по дереву)

Дополнительно: строка глобального поиска сверху (см. 5.4), доступная
с любого экрана.

### 5.2. Экран "Поиск по коду ошибки"

- Поле ввода кода (например `P000A00`), с клавиатурой, оптимизированной
  под буквенно-цифровой ввод.
- При вводе — можно рассмотреть автодополнение/подсказки по мере
  набора (не обязательно для MVP, опционально).
- После ввода полного кода и нажатия "Найти":
  - Если код найден: показать `description_ru` (или `description_en`,
    если перевод ещё не готов), `ecu`, `category`.
  - Под расшифровкой — список документов, у которых `ecu_tags`
    содержит найденный ECU (запрос к `documents` по полю `ecu_tags`,
    `array-contains`).
  - Каждый документ в списке — кликабельная карточка (название,
    размер файла, количество страниц) → открывает экран просмотра PDF.
  - Если код не найден: понятное сообщение ("Код не найден. Проверьте
    правильность ввода."), без технических деталей об устройстве базы.

### 5.3. Экран "Категории документации"

- Древовидная навигация по `categories` (используя `parent_path` для
  построения иерархии), начиная с категорий верхнего уровня
  (01_Engine, 02_Transmission и т.д.).
- На конечном уровне категории — список документов этой категории
  (запрос к `documents` по полю `category`).
- Клик по документу → экран просмотра PDF.

### 5.4. Полнотекстовый поиск (глобальный)

- Поле ввода произвольного текста (например "ремень ГРМ").
- Поиск идёт по полю `full_text` документов (реализация — см. раздел 6).
- Результат — список документов, где найдено совпадение, с коротким
  фрагментом текста вокруг найденного слова (для контекста, по
  аналогии с обычной поисковой выдачей).
- Клик по результату → открывает экран просмотра PDF (в идеале сразу
  на странице с найденным текстом — если удаётся определить номер
  страницы, см. 6.3).

### 5.5. Экран просмотра документа

- Отображение PDF-файла (открывается из Firebase Storage).
- Стандартные функции просмотра: пролистывание страниц, зум,
  переход на конкретную страницу.
- Кнопка "Скачать" / "Открыть в другом приложении" (на случай, если
  встроенный просмотрщик не устраивает пользователя или нужна
  offline-копия).
- Индикатор загрузки, пока файл скачивается (файлы могут быть
  крупными, важно не давать приложению выглядеть зависшим).

## 6. Поиск — детали реализации

### 6.1. Поиск по коду ошибки
Прямой запрос к Firestore по составному ID документа
(`{model}_{ecu}_{dtc_code}`) либо, если ECU неизвестен пользователю
заранее, запрос по полю `dtc_code` с фильтром по `model`. Быстро,
просто, не требует дополнительной инфраструктуры.

### 6.2. Поиск по категориям
Обычные запросы к Firestore по полям `category` / `parent_path`.
Простая иерархическая навигация, без особых сложностей.

### 6.3. Полнотекстовый поиск — важное техническое ограничение

Firestore **не поддерживает полнотекстовый поиск "из коробки"** (нет
аналога SQL `LIKE '%...%'` или полнотекстовых индексов). Для MVP
предлагается следующий практичный подход:

- Поскольку документов всего 21 (для J7), объём текста для поиска
  небольшой. Вместо серверного полнotекстового индекса — **загружать
  весь текстовый корпус (или его значимую часть) на клиент один раз
  при первом запуске / обновлении данных, и делать поиск локально**
  (простое совпадение подстроки/токенов в памяти устройства).
- Практическая реализация: хранить `full_text` не как поле в самом
  документе Firestore (это может раздуть размер документа и стоимость
  чтений), а как отдельный `.txt` файл в Firebase Storage рядом с
  PDF (например `j7/01_Engine/HFC4GB2.4D/maintenance.txt`), путь к
  которому указан в поле `text_path` документа `documents`. Клиент
  скачивает нужные текстовые файлы (или все — при разумном общем
  объёме) и ищет по ним локально.
- Такой подход работает для MVP на 21 документе. Если в будущем
  (при расширении на 15 моделей) объём текста станет большим —
  потребуется пересмотреть архитектуру (например Algolia, Typesense,
  или серверная функция Cloud Functions с индексом) — но это
  сознательно выносится за рамки MVP.
- Поиск должен быть нечувствителен к регистру, желательно —
  устойчив к базовым словоформам (простой поиск по подстроке без
  морфологии — приемлемо для MVP; полноценный морфологический поиск —
  избыточен на этом этапе).

## 8. Критерии готовности MVP

MVP считается готовым, когда механик может, используя только
приложение на своём телефоне:

1. Ввести код ошибки J7 → увидеть его расшифровку на русском (или
   английском, если перевод не готов) и список релевантных документов.
2. Открыть любой из 21 документов J7 и пролистать/прочитать его
   прямо в приложении, без необходимости скачивать сторонний
   просмотрщик.
3. Найти документ через дерево категорий, не зная кода ошибки.
4. Ввести произвольную поисковую фразу (например "замена масла") и
   получить список документов, где эта фраза встречается, с коротким
   контекстным фрагментом.
5. Всё это должно работать стабильно на обычном 4G/Wi-Fi соединении,
   без критичных задержек (загрузка документа — разумное время с
   явным индикатором прогресса).
